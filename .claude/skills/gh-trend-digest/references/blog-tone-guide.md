# 한국어 친근 블로그 톤 가이드

`content-writer` 에이전트가 따라야 할 글쓰기 톤 규칙. 백서가 아니라 **친한 동료 개발자에게 "이거 봤어?" 하고 소개하는** 글이 목표다.

## 톤의 4원칙

### 1. 사람 냄새가 나야 한다

- 글의 화자가 살아 있음을 드러낸다: "나는 ~", "내 생각엔 ~", "직접 돌려봤더니 ~"
- 감탄·의문·놀라움을 숨기지 않는다: "이 부분이 진짜 흥미로운데,", "이게 왜 되지?"
- 단, 과한 이모지·!! 남발은 금지. 한 글에 이모지 0~2개가 적정.

### 2. 호흡을 다양화한다

- 같은 종결어미를 4문장 이상 연속하지 않는다. "~다. ~다. ~다. ~다." → 끊긴다.
- 짧은 문장과 긴 문장을 교차한다.
- 한 단락을 7~8문장 넘기지 말 것. 길어지면 단락 나누기.

**나쁜 예:**
> 이 라이브러리는 빠른 성능을 제공한다. 메모리 사용량도 적다. 다양한 언어를 지원한다. 문서도 잘 되어 있다.

**좋은 예:**
> 빠르다. 그런데 단순히 빠른 게 아니라, 메모리도 거의 안 먹는다. 거기에 지원 언어 목록까지 보면 — 이게 한 사람이 만든 거라고? 싶을 정도. 문서까지 친절하다는 게 좀 비현실적이다.

### 3. 백서 톤 금지

다음 표현은 가능하면 다른 말로 바꾼다:

| 백서 톤 (피할 것) | 블로그 톤 (권장) |
|------------------|-----------------|
| ~을(를) 제공한다 | ~할 수 있다 / ~가 들어 있다 |
| ~을(를) 지원한다 | ~도 된다 / ~까지 된다 |
| 본 프로젝트는 | 이 프로젝트는 / 이건 |
| 사용자는 ~할 수 있습니다 | 그냥 ~하면 된다 |
| 핵심 기능은 다음과 같다: | 뭘 해주냐면, |
| 도입 시 고려사항 | 쓰기 전에 알아둘 것 |

### 4. 의견은 의견으로 드러낸다

- "X가 좋다"는 단정 대신 "X가 좋다고 느꼈다", "X 부분이 마음에 들었다"
- 비판도 마찬가지. "Y가 나쁘다" → "Y는 좀 아쉬웠다", "Y에서 막혔다"
- 사실과 의견을 섞지 마라. 사실은 자료 기반, 의견은 명시적 의견으로.

## 글 구조 패턴

### 도입은 개인적으로

데이터/통계로 시작하지 말고, **"내가 이걸 어떻게 만났고 왜 흥미로웠는지"**로 시작. 한 문단, 길어도 두 문단.

**나쁜 예:**
> X는 2024년 5월에 출시된 라이브러리로, 출시 6개월 만에 GitHub 스타 30,000개를 돌파한 트렌딩 프로젝트다.

**좋은 예:**
> 며칠 전 트위터에서 누가 X를 깔아보라길래 깔았다. 처음엔 "또 비슷한 거겠지" 했는데, README 첫 화면의 한 줄 — `npm i x` 하면 끝 — 에 좀 끌렸다. 직접 30분 돌려보고 나서 글을 쓰기로 했다.

### 코드 스니펫은 "보여주고 → 설명"

```markdown
한 가지 인상적이었던 건 핸들러 디스패치 방식이다.

\`\`\`typescript
// src/router/dispatch.ts:42-58
async function dispatch(req: Request, route: Route) {
  for (const mw of route.middlewares) {
    const result = await mw(req);
    if (result.terminate) return result.response;
  }
  return route.handler(req);
}
\`\`\`

미들웨어가 `terminate: true`를 반환하면 즉시 응답하고 핸들러로 안 간다. Express의 `next()` 패턴 대신 return 값으로 흐름 제어를 한 거다. 명시적이라 디버깅하기 좋아 보였다.
```

- 스니펫 위 2~3문장으로 "왜 보여주는지" 깔고, 아래 2~5문장으로 해석
- 스니펫만 던지지 말 것

### 비교 섹션은 표 + 산문

표만 두면 건조하다. 표 다음에 한 단락으로 "결국 이걸 쓸 이유"를 풀어낸다.

### 마무리는 짧고 솔직하게

"훌륭한 프로젝트입니다" 같은 결론 금지. 본인 느낌·다음 행동·기대를 한두 문장으로.

**좋은 예:**
> 당장 회사 프로젝트에 넣지는 못하겠지만, 다음 사이드 프로젝트엔 한번 써볼 생각이다. 0.x 버전이 1.0이 되는 걸 지켜보고 싶어졌다.

## 자주 하는 실수

- **자료를 그대로 옮긴다** — 자료는 자료고, 글은 글이다. 자료에서 5개 사실을 가져왔다면, 그걸 너의 흐름으로 재배치한다.
- **모든 기능을 다 나열한다** — 인상적인 것 3~5개에 집중. 나머지는 "이 외에도 X, Y, Z가 있다"로 한 줄 처리.
- **부풀리기로 분량 채우기** — 같은 말 반복하면 리뷰어가 잡아낸다. 분량이 안 나오면 자료가 부족한 거다.
- **인용 출처 누락** — 블로그 톤이라도 출처는 살린다. 본문에 자연스럽게 녹이거나 마지막에 "참고" 섹션으로.

## 셀프 체크

글을 다 쓰고 다음 질문에 답해본다:

1. 이 글에서 가장 사람 냄새 나는 문장 3개를 꼽을 수 있나?
2. "~을 제공한다" / "~을 지원한다" 문장이 5개 이상인가? → 줄여라
3. 같은 종결어미가 연속 4문장 이상 나오는 구간이 있나?
4. 도입이 데이터로 시작하나? → 다시 써라
5. 마무리가 "훌륭한 프로젝트다"인가? → 다시 써라
